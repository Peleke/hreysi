# The tool built to detect silent failure was failing silently

## What this session set out to do

Install LifeOS, then talk about fixing the content pipeline. It became: install
LifeOS (finding four of its bugs on the way), then discover that the *content
pipeline's own capture layer* had the exact failure mode I'd just been swatting in
LifeOS — a tool reporting success while doing nothing. The day had a spine before it
had a name, and the name turned out to be the same one four times over.

## The day capture was dead and the doctor said fine

The tell was a `.git/hooks/post-commit: /tmp/hreysi-build/hreysi: No such file or
directory` scrolling past during a commit. hreysi's own hook pointed at a `go build`
artifact from days earlier, long deleted. Every commit since had run that line, hit
`|| true`, and journaled nothing. Dogfooding had been dead and I hadn't noticed
because *nothing tells you* — that's what `|| true` buys you.

So I ran `hreysi doctor`, the command whose entire reason to exist is catching this.
It said **"capture is live — every commit will be journaled."** Green. Exit 0.

Three bugs were holding hands. `doctor` proved someone had once run `init` here
(`strings.Contains(hook, marker)`) and called that "wired" — it never checked the
binary resolved. Its verdict was a hand-maintained `exists && wired && executable`
that didn't include the check I was about to add, so it would print "live" directly
over a `✗`. And `init` itself, told to repair, saw its own marker and said
"up-to-date" — meaning doctor's printed advice, *rerun init*, was a lie too.

The fix that felt right: stop letting the verdict restate a subset of the checks.
`Healthy` is now `true` unless some check said `fail`. Any future check is
automatically load-bearing on the verdict — you can't add one and forget to wire it
into the summary, because there's nothing to wire.

## The tests were in on it

The part that stung: the suite was green, and the suite was *wrong*. Three fixtures
did `Init(dir, "/usr/local/bin/hreysi")` — a path absent on most machines — and
asserted the result was healthy. They only ever passed because the old `doctor`
never looked. The tests were certifying the bug.

CI made the point louder than I could. My machine stayed green after the fix because
one fixture used the bare name `hreysi`, which resolves on my PATH (brew). CI has no
hreysi, so it went red on exactly the case I couldn't see locally. I'd have shipped
it. The lesson isn't "run CI" — it's that a fixture asserting a resource *it didn't
create* exists is lying, and the lie is invisible until the environment changes.

## Mirror: one-way, and afraid of the vault

With capture alive again, the pipeline gap: hreysi writes `buildlog/` per repo, but
the Obsidian vault holds a *separate* 34-entry corpus from the deprecated
buildlog-template, and nothing bridged them. No consumer could see a week of work,
only a week of work in one repo. A digest can't find a throughline it can't see.

`hreysi mirror` copies expanded entries into the vault. The whole design pressure was
one sentence: **do not eat the 34.** Every file mirror writes carries `source: hreysi`
in frontmatter; any destination lacking that marker is somebody else's and mirror
refuses it, loudly, and moves on. I wrote the clobber-refusal test first and made it
the meanest one in the file — a foreign file at the destination must come back
byte-identical. Ran it against the real vault: 33 → 34, today's un-expanded spine
correctly skipped, all 33 legacy entries untouched.

Kept it dependency-free — the vault path is a one-line file, not TOML — because
hreysi's zero-dependency `go.mod` is a feature and a parser isn't worth spending it.

## The thread that ran through everything

Four bugs this session, one shape. hreysi's doctor said live while capture was dead.
LifeOS's Pulse plist ran with an unsubstituted `__BUN_PATH__` and exited 78 while
healthz stayed quiet. LifeOS's memory hooks wrote to a literal `$HOME/` dir for weeks
while the dashboard said "no data." linwheel's `qortexFetch` swallowed every error and
returned `null`, which the caller read as "not configured." None of them *broke*. They
all *succeeded quietly at the wrong thing*.

It seems to be the signature failure of well-tested, LLM-generated code: the happy
path is covered, the code is confident, and the silence where an error should be is
the bug. `|| true` is the whole genre in three characters.

## Improvements

### Architectural
- A health/verdict field must be derived from the checks it summarizes, never a
  hand-maintained subset. The subset drifts; the derivation can't.
- When writing into a directory you don't own, gate on a positive provenance mark you
  control, not on a naming convention or a directory boundary. Absence of the mark is
  the safe default: refuse.

### Process
- Write the "must never happen" test first and make it the strictest in the file. For
  mirror that was "a foreign file comes back byte-identical."
- A test fixture that asserts a resource it didn't create is a false green waiting for
  an environment change. If a test needs a binary/file/service to exist, it should
  create it.

### Tooling
- `|| true` on a hook or cron line converts every downstream failure into silence.
  Anything guarded that way needs an independent liveness check that resolves the real
  target, because the guard guarantees you'll never hear about it otherwise.

### Gotchas
- Green locally, red in CI, is often PATH: a fixture passing a bare command name
  resolves against the dev's installed tools and vanishes in a clean environment.
- Claude Code does not shell-expand env values — a literal `$HOME` in a settings `env`
  block reaches hooks verbatim and any that read it raw build a path under the cwd.
